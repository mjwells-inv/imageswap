Images and Placeholders Guide

What do you see where there should be an image?

After the page has finished loading, what do you see? If there are mixed results, you should usually be able to base your approach on the least functional images.


Nothing: Not even a broken image icon. This can happen when an image is set as the background. Since there is nothing to right click on, it is harder to find the URL in these cases.

Note: this option's results are all the same as the results for "Broken image icon." Only some minor wording is different.


Broken image icon: There might be an empty frame with the icon inside or just the icon. The icon might have the name of the image next to it or might be all alone.

Edge case note: Image host error images like these (with text saying an image doesn't exist or can't be found) can also be counted under "Broken image icon."


Obvious placeholder: This would be something like a box with a word in the middle. If in doubt, count it as an actual image.


Actual image: Any image that successfully loads and isn't an obvious placeholder.



What is the image URL?

To find out, right click on the image or broken image icon and use "inspect" or "copy image address."

If there is nothing to right-click on, you can use "inspect" to look for a URL within the style header (click the arrow next to "head" and then "style"). If there's nothing obvious there, try viewing the source code (ctrl+u) and using ctrl+f to search for some key terms. If you search for "http" you'll find any URLs; just be aware that they might not always be for images. Also, if you search for "data:image" it should turn up any attempt to use a Base64 encoded image.


Note: If you use Michael Wells's Media Replacer, it will change the URLs used. You need to answer this question based on the URLs in the original version of the webpage. The Media Replacer will also tell you the URLs that were replaced from the original page.


URL is itself a placeholder: This is not an attempt to insert a placeholder image. This is when the webpage code includes a made-up file name or URL (like “photo1.jpg” or “https://mybusiness.com/mylogo.png”) that isn't supposed to work. It's just supposed to show the user where to put the real URL or file name. If a URL is clearly intended to be a made-up placeholder, treat it like a placeholder URL, even if the URL/domain also happens to be real (like the "mybusiness.com" example above, or any time an "example.com" URL is used).

Note:A local file name like "photo1.jpg" will show up as "photo1.jpg" when you use "inspect" or look in the source code. However, if you use "copy image address," it will look like "https://featherstorageprod-xxxxx.azurefd.net/assets/photo1.jpg". So if you see a "featherstorage" URL, a placeholder file name was probably used.


Base64 string: Encoding in Base64 is a way of including an image directly as part of the html. Where the URL would normally be, instead there would be something like "url('data:image/jpeg;base64," followed by a long string of jumbled up characters.


Link to image placeholder website: Choose this if the URL links to a website that is primarily and explicitly intended to provide placeholder images. The option applies for both working sites and sites that were known placeholder image providers but are no longer available/reliable.

Some common active placeholder image websites:

  • placehold.co
  • loremflickr.com
  • picsum.photos (Lorem Picsum)
  • placehold.jp
  • fakeimg.pl
  • images.placeholders.dev
  • dummyimage.com
  • placebear.com
  • i.pravatar.cc (Profile/avatar pictures)
Some known dead (or dying) placeholder image websites:
  • via.placeholder.com
  • placekitten.com
  • placedog.com

Link to any other website / Other: Choose this if the URL links to any website that is not primarily and explicitly intended to provide placeholder images. This includes free photo/media websites such as commons.wikimedia.org, unsplash.com, and pixabay.com because they are not primarily intended for providing placeholders.


The "/ Other" part: The Media Replacer can identify some additional fringe cases and give instructions about how they should be classified. However, if you encounter a situation that doesn't seem to fit anywhere else, you can choose this as a fallback option.



Does the completion text say that placeholders were used?

We have been told that any time placeholders are used, the completion text must explain to the user that placeholders were used. The text does not need to use the word "placeholder." Any statement that an image URL or file name can or must be replaced by the user should count as a "yes," and Ctrl+f searching for the word "replace" is often the quickest way to find placeholder explanations.


Yes: We are not currently evaluating the quality or accuracy of the completion text, so choose "yes" if the completion text states that placeholders were used for images, even if part of the statement (such as the source of the placeholder images) isn't accurate.


No: Choose this if the text doesn't state that the user can or should replace the included images with their own. If the completion includes only a webpage link and no additional text, that always counts as a "no."



General guidelines for failure to acknowledge placeholders

Failure to acknowledge placeholders in the completion text should be considered a minor flaw. It may negatively affect functionality scores above 4, but it should never push a 4 down to a 3 or have any effect on scores below 4.
(However, it can always be used as an overall rating tie-breaker if there aren’t more significant differences to consider.)
